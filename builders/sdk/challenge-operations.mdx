The challenge system runs as a policy program (`policy_challenges`) that doubles as the prop-firm engine. Templates encode the dual-track economics.

## purchaseChallenge

```typescript
const { challenge, signature } = await hyro.purchaseChallenge({
  templateId,           // which challenge template (size + rules)
  payer: wallet.publicKey,
});
```

When called, the evaluation fee is recorded on-chain, lands in the fee vault, and the rulebook version is pinned to the challenge.

## getChallenge

```typescript
const c = await hyro.getChallenge(challengeId);

console.log(c.status);          // initialized | active | tier1 | tier2 | failed
console.log(c.startingBalance);
console.log(c.latestBalance);
console.log(c.tierUnlockedAt);  // timestamp when +15% crossed, or null
console.log(c.vaultOpened);     // whether the real-capital vault has opened
console.log(c.payoutPaid);
```

## getChallengeTemplate

```typescript
const t = await hyro.getChallengeTemplate(templateId);

console.log(t.stageType);        // evaluation | funded_tier1 | funded_tier2
console.log(t.startingDeposit);
console.log(t.dailyDrawdown);    // 4%
console.log(t.maximumLoss);      // 6% trailing
console.log(t.profitTarget);     // 10%
console.log(t.tierThreshold);    // +15% = unlock dual-track
console.log(t.tier1Split);       // 80% trader
console.log(t.tier2Split);       // 50% on incremental
```

## The dual-track in template terms

| Field | Meaning |
|---|---|
| `tier_threshold` | +15% — when Tier 2 unlocks **and** the real-capital vault opens |
| `tier1_split` | 80% trader (0 → +15%) |
| `tier2_split` | 50% trader on incremental (+15% onward) |
| `minimum_trading_days` | 5 |
| `profit_target` | 10% |
